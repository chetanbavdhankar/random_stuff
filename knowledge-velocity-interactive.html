Step 2 — The insight

Expert knowledge is not missing. It is written down daily and discarded. Every L3 return comment is a senior investigator teaching. Every QC repair note is a quality standard being articulated. Today each teaches one analyst, once. Reading thousands of them in aggregate used to cost analyst-weeks — LLMs drop that cost to near zero. The entire opportunity is noticing this.

Clustering protocol (technical)

Corpus prep. Export comments with case metadata: journey, case type, analyst tenure band (never named analysts at this stage), return/repair date, outcome. Strip customer PII before any LLM contact — names, account numbers, amounts bucketed.

Clustering. Two-pass approach inside the tenancy ceiling: (1) per-comment classification prompt producing a short defect phrase + evidence-type label; (2) group the phrases and merge near-duplicates. Batch through Copilot Studio / AI Builder prompt actions; volume is thousands of short texts, well within limits if batched.

Adjudication. Present candidate categories with 5 example comments each. SMEs merge, rename, reject. Rule: every surviving code must be phrased as a checkable evidence requirement, not a judgment ("SoF rationale present in narrative" — not "weak analysis").

Validation. Double-code a 100-comment sample: LLM assigns codes, one SME independently assigns codes, measure agreement. Below ~80% agreement → refine code definitions and repeat. This number goes on a slide; it is what makes the taxonomy defensible when L3 disputes a flag.

Costing model — from counts to FTE

Per defect code: frequency × unit_cost, where unit cost = average rework hours (analyst) + L3 review minutes + queue days converted at loaded cost. Get unit costs from team leads as ranges, not precision theatre — present low/high bands. The Pareto in FTE is what moves leadership; counts only inform.

Why SME sign-off is non-negotiable

A taxonomy L3 didn't sign is a taxonomy L3 will dispute the first time the agent flags one of their patterns. The LLM does the drudgery; the SMEs supply the authority. The 2-hour workshop is the scarcest resource in the whole plan — commit it before the pitch or the timeline is fiction.

The Pre-Flight Check — try it

An analyst invokes the agent in Teams before escalating to L3 or submitting to QC. It checks whether required evidence is present — never whether the conclusion is right. Simulated below with a shortened synthetic case.

PRE-FLIGHT CHECK · UMR → L3 escalation gatecase UMR-2026-0417 (synthetic)

Analyst narrative (excerpt): Customer received 14 inbound transfers totalling AUD 182k over 3 weeks from 9 distinct originators, inconsistent with stated salary profile. Funds moved to a newly opened account at another institution within 48h of receipt. Pattern consistent with mule activity. Recommend escalation to L3.

Rubric: signed-off defect codes · grounding: SharePoint procedure library
● HOLD — 2 evidence gaps before escalation
CODE 01 · SOURCE-OF-FUNDS RATIONALE ABSENT
Narrative asserts inconsistency with salary profile but does not evidence the profile itself (declared occupation, expected turnover, KYC refresh date). L3 returned 34% of Q1 cases citing this gap.
→ Procedure: TM-AU-INV-012 §4.2 "Customer profile baseline"
CODE 02 · COUNTERPARTY SCREENING NOT EVIDENCED
9 originators referenced; no record of screening outcome or internal-account linkage check in the narrative or attachments.
→ Procedure: TM-AU-INV-012 §5.1 "Originator due diligence"
CODE 03 · TYPOLOGY MAPPING — PRESENT
Mule-activity typology cited and consistent with the transaction pattern described.

Gap list + traffic light. Never a numeric score. The analyst decides — nothing is auto-approved, QC sampling continues unchanged.

Architecture (technical) 
Teams (analyst) ─► Copilot Studio agent
                       │  topic: pre-flight check
                       │  input: case narrative (v1: analyst paste;
                       │         v2: Power Automate pull from case mgmt if API exists)
                       ├─► generative answers grounded on:
                       │     SharePoint procedure library (global-standard docs)
                       │     defect-code rubric table (Dataverse)
                       ├─► per-code evidence-presence check (prompt action per code)
                       └─► Dataverse log: case ref · codes flagged ·
                           analyst action · downstream L3/QC outcome

Design rules (hard): gap list + traffic light, never scores (scores get gamed and misread as decisions). Evidence-presence checks, never conclusion checks (kills the gaming vector and keeps Model Risk classification low). Human always decides.

Integration honesty: if case management exposes nothing Power Automate can touch, v1 is analyst-initiated paste. Ugly, ships, pilots fine at 10 analysts — stated openly rather than hand-waving an unconfirmed integration.

Retro-test protocol — the backtest before go-live

Take 40–50 cases L3 actually returned last quarter plus a control set of accepted cases. Run the rubric against the pre-escalation narratives. Report: hit rate (did the gate flag the real return reason?) and false-positive rate on the control set. This converts the pitch from "here's a chatbot" to "here's measured performance against historical ground truth" — and it pre-answers Model Risk. Run it even if the agent build is half-finished at demo time; the numbers are the demo.

Copilot Studio constraints to budget for

Grounding-size limits on SharePoint sources (chunk the procedure library, one doc per code family). Generative-answer variance (pin the rubric checks to prompt actions with structured output, keep free generation only for citations). AI Builder credit consumption at pilot volume — budget credits explicitly in the proposal; a pilot that dies on licensing in month four is an avoidable embarrassment.

Layer 3 · Weeks 8–26 (overlapping)

Instrument: close the loop at source

The first pass is a mining exercise. Every pass after is a query — because the process now records its own failure reasons in structured form, at the moment they occur, at a cost of two seconds per analyst.

The two-second change

Add a reason-code dropdown (the signed-off taxonomy) at exactly two points: the L3 return action and the QC repair action. Free text stays available for nuance; the code is mandatory. If the case-management UI owner can't slot the change, a SharePoint form intercepting the same workflow step is the workaround — name the owner, don't hand-wave.

Risks, pre-conceded

R1 Corpus too terse

If returns say "insufficient evidence" and stop, the taxonomy is 4 codes and dead. Gate: sample 50 comments before committing (1 hour). Fallback: synthetic corpus labeled as methodology demonstration, hit-rate claims removed; framing survives on the interview document alone.

R2 Gaming the checker

Analysts optimize to satisfy the gate rather than investigate. Mitigated structurally: rubric checks evidence presence not conclusions; QC sampling continues unchanged; nothing auto-approved.

R3 Model Risk classification

Decision-support, human decision-maker, no customer impact, no regulator-facing artifact generated → should classify low-tier, but ING's framework decides, not us. Engagement booked week 2, on the slide — costs nothing, signals shipping experience.

R4 Copilot Studio limits

Grounding size, generative variance, AI Builder credit burn at volume. Credits budgeted explicitly; structured prompt actions over free generation for the rubric checks.

R5 SME attention

The true scarce resource: 2 hours of the L3 lead. Committed before the pitch, or the timeline is fiction.

R6 Case content reaching the agent

If case management exposes no automatable surface, v1 is analyst paste. Ugly, ships, pilots fine at 10 analysts. Stated openly.
