What the methodology is built on

Three published methods, one investigation system.

Each component of this design is taken from a peer-reviewed or preprint method that has been tested on public AML benchmarks — not invented for this deck. The three papers below respectively supply the graph representation, the agentic architecture, and the evidence & verification discipline.

[P1]

In-Context Learning of LLMs for Money Laundering Detection in Financial Graphs

Pirmorad (Vanguard) · arXiv:2507.14785

Shows that an LLM can perform investigator-style reasoning when transactions are represented as a financial knowledge graph and each case is presented as a k-hop subgraph serialized to structured text, conditioned by few-shot examples of canonical laundering topologies.

  • Edge-centric framing: the alerted transaction is the anchor
  • Serialization preserves topology, direction, timestamps, values
  • Eight canonical patterns as reasoning demonstrations
  • Finding: LLM reasoning is strongest on structured patterns; recommends hybrid triage (light classifier first, LLM for borderline cases)
[P2]

Co-Investigator AI: Agentic AI for Trustworthy AML Compliance Narratives

Naik et al. · arXiv:2509.08380

An industrial multi-agent framework for SAR narrative generation, validated by six expert AML investigators. Supplies the agent decomposition and privacy handling.

  • Data Ingestion & Structuring layer feeding all agents
  • AI-Privacy Guard: anonymize sensitive entities before any external model call, unmask on return
  • Crime-type detection: risk-indicator extraction + tree-ensemble classifiers → ranked typologies with confidence
  • Planning agent dynamically spawns typology-specific agents
  • Compliance Validation Agent (Agent-as-a-Judge) + golden-dataset evaluation
[P3]

Explainable AML Triage with LLMs: Evidence Retrieval and Counterfactual Checks

Torres, Cheng & Hu · arXiv:2604.19755

Formalizes triage as an evidence-constrained decision process and shows on public synthetic benchmarks that grounding + verification beats both classical ML and free LLM generation. Supplies the evidence model, output contract, verifier, and counterfactual layer.

  • Evidence pool ℰ = {policy, KYC, triggers, transactions, prior cases}, each item with stable ID, timestamp, source type, ACL tag
  • Hybrid retrieval → compact evidence bundle B(a) with per-type quotas
  • Output contract: disposition, confidence, cited rationale, supporting vs contradicting vs missing evidence, next actions
  • Verify-and-repair loop; decision & evidence counterfactual checks

Why this combination. [P3] demonstrates empirically that an LLM alone is a weak triage predictor, but the same LLM inside an evidence-grounded, verified workflow outperforms rules, gradient-boosted and graph-ML baselines while nearly eliminating fabricated citations and numerical errors. [P1] supplies the representation that makes graph-shaped laundering behavior legible to the model, and [P2] supplies the agent decomposition and privacy layer proven with practitioner evaluation. The methodology below is the composition of the three.

The core of the system

D0–D7: from Excel files to an evidence bundle.

Eight deterministic stages transform raw exports — transaction lists, KYC extracts, alert dumps — into the representations the papers require: a financial knowledge graph, a serialized case subgraph, and a citable evidence bundle. Every stage is code, not model output: reproducible, testable, and versioned. All examples below are structural templates with placeholder tokens, never real or invented values.

D7+ The eight canonical laundering topologies used as few-shot demonstrations

From [P1] §2.3 — the exemplar library is built from these structural patterns, each paired with a serialized subgraph in the D5 format plus an analyst-style rationale, alongside benign counter-examples so the model learns to distinguish, not just to accuse.

Reasoning under constraint

The agent never writes freely — it fills a contract.

Following [P3], triage is an evidence-constrained decision process: the model may only assert what it can cite from the bundle, its output must follow a fixed schema, and everything it produces is machine-verified and counterfactually stress-tested before an investigator sees it.

A.2 The output contract

Exactly the structured triage record of [P3] §2.3: not an essay, a schema. Two properties matter most for compliance review: every rationale paragraph must carry at least one evidence citation, and the record must explicitly separate what supports the concern, what contradicts it, and what is missing — because in AML, absence of corroboration is decision-relevant information.

triage_record — output schema (types, not values)
{
  "disposition":     "dismiss | monitor | escalate",
  "confidence":      "calibrated score",          // [P3] §2.3: u
  "typologies":      "ranked list + scores",       // routes workflow
  "rationale":       "paragraphs; each ≥1 citation to an evidence_id",
  "evidence": {
    "supporting":    "[evidence_id, …]",
    "contradicting": "[evidence_id, …]",
    "missing":       "what was sought and not found"  // explicit unknowns
  },
  "next_actions":    "RFI / expand lookback / check related accounts",
  "counterfactuals": "validated minimal flip conditions"   // [P3] §3.4
}

A.3 Verification — two independent layers

Layer 1 · verify-and-repair · [P3] §3.3

Provenance & consistency verifier

Provenance integrity: every cited evidence_id must exist in the supplied bundle — fabricated citations are rejected outright.

Numerical & temporal consistency: every amount, date, ordering claim and threshold comparison in the rationale is checked against the structured fields of the cited item.

Repair loop: a failed check returns targeted feedback (e.g. "cited ID absent from bundle", "stated total conflicts with evidence E_T#") and triggers bounded regeneration — the draft cannot reach the investigator until checks pass.

Layer 2 · counterfactual checks · [P3] §3.4

Faithfulness stress tests

Decision counterfactuals: propose minimal, plausible edits to the case (toggle a claimed key indicator, adjust a counterparty risk tier, narrow the window, remove one graph path) within an edit budget and plausibility rules; a policy-approximate validator confirms the disposition flips as the rationale claims it should.

Evidence counterfactuals: remove a critical cited item and re-run — the model must stop citing it and adjust its conclusion coherently. This is the direct test against "decorative citations": explanations that sound grounded but don't actually drive the decision.

Why an LLM at all, then? Because [P3]'s ablations show the ordering clearly: the LLM alone is the weakest triage method they tested — worse than gradient boosting — but the LLM inside this evidence-constrained, verified loop is the strongest, beating rules, tabular ML and graph-ML baselines on both ranking quality and escalate-F1 while raising citation validity and evidence support to near-perfect levels. The value is not free-form reasoning; it is structured decision support over explicit evidence.

A.4 Cost & routing

Per [P1] §4 and [P3] §7.5: not every alert needs the full pipeline. A lightweight scorer (the D4 features feeding a gradient-boosted or graph model) triages the whole queue; clear-cut alerts get the fast path, and only the ambiguous or high-risk band is routed through subgraph serialization, full agent investigation, verification and counterfactual checks. Repair iterations and counterfactual budget are capped per alert. This keeps latency and inference cost proportional to case difficulty.

Proving it, honestly

Measured the way the papers measure it.

The evaluation plan adopts the metric suite of [P3] §5 and the golden-dataset protocol of [P2] §7, instantiated first on public synthetic benchmarks and our own backtesting archive, then in shadow mode. We report nothing we have not measured — the only numbers in this section are the papers' published results, clearly attributed.

E.2 What the published benchmarks already show ([P3] Table 1)

On SynthAML / AMLSim-based benchmarks, the paper reports its full pipeline (retrieval grounding + counterfactual checks) at PR-AUC 0.75 and Escalate-F1 0.62 — above graph-ML (0.69 / 0.56), tabular GBDT (0.63 / 0.51), rules (0.41 / 0.29) and the LLM-only baseline (0.52 / 0.41) — with citation validity 0.98, evidence support 0.88, counterfactual faithfulness 0.76, and numerical-inconsistency and policy-hallucination rates reduced to 0.03 and 0.02. These are the authors' published results on public synthetic data, cited here as external evidence for the architecture choice — not our system's performance, which will be measured on our own cases before any claim is made.

E.3 Protocol discipline

[P3] §4.4

Time-aware splits

Train on earlier alerts, validate and test strictly on later ones — matching deployment reality and avoiding the optimism of random splits on time-evolving transaction data.

[P3] §4.4

Retrieval leakage control

Case memory indexes training-period cases only; no test labels or post-decision artifacts are ever retrievable. In RAG systems retrieval is otherwise an inadvertent label channel.

[P2] §7

Golden datasets

Expert-annotated adjudicated cases as the benchmark; hybrid scoring combining rule-based factual checks on the intro elements with weighted semantic assessment of the narrative body.

E.4 Staged rollout ([P3] §7.3)

stage 1
Synthetic benchmarks

Functional correctness and failure-mode analysis on public AML simulators + our synthetic corpus.

stage 2
Historical backtest

Full metric suite on adjudicated closed cases from the backtesting archive, per typology and per segment.

stage 3
Shadow mode

Agent drafts in parallel; investigators decide independently; agreement and time measured before any output is shown pre-decision.

stage 4
Guarded assist

Drafts shown with mandatory human sign-off, sampled QC, drift monitoring, capped repair & counterfactual budgets.

Governance posture. The D0–D7 pipeline and both verifiers are deterministic, versioned code — validatable under standard model-risk treatment. The LLM layer is decision support: it never dispositions autonomously, its factual scope is bounded by the retrieved bundle ([P3] R1–R4), its outputs are machine-verified, and every retrieval, tool call, verifier result and counterfactual outcome is logged as the audit trail [P3] §7.2 describes for model-risk management.
