Threshold headroom — design and sandbox

A KRI that detects alert definitions where the threshold can be raised, quantifies the efficiency prize, and states the detection risk explicitly. Every control below is live and linked: data shape → headroom → reclaimed volume → guardrails → decision.

1Objective

Identify alert definitions (ADs) where the monitoring threshold sits well below the lowest productive alert, so that raising it eliminates the non-productive alerts in between — reclaiming analyst hours with a known, controlled effect on detection.

The decision variable is the lower tail of the productive distribution: how close the nearest productive value sits to the threshold. Everything in this design is built to measure that tail, price the gap below it, and gate the decision with explicit risk controls.

2The design — three numbers per AD

All alert values are first normalized to the threshold, r = value / T, so every AD, entity, and parameter type (amount, count, ratio) lives on one comparable axis where r = 1 is the threshold.

H = q_α( productive r-values ) HEADROOM — the α-quantile of normalized productives. The maximum safe threshold multiplier. α = 0 → H = lowest productive (zero tolerated loss). R = #{ FP alerts with r < H } RECLAIMED — non-productive alerts that would never have been generated under the raised threshold. The efficiency prize; converts directly to hours. P = #{ productives with r < H } AT RISK — observed productives sacrificed at the chosen α. Zero by construction at α = 0. The number Model Risk will ask for. FIRE ⇔ H ≥ h_min AND R ≥ r_min AND guardrails pass evaluated on a pooled rolling window (default 4 quarters).
Risk appetite α
Tolerated fraction of observed productives below the raised threshold. A business/compliance dial, set per risk tier — not a statistical constant.
h_min
Minimum headroom worth acting on. Below ~1.25× a raise buys little and leaves no safety margin.
r_min
Minimum reclaimed volume. Set so the saved hours clear the cost of running an ATL/BTL tuning cycle.
Pooled window
All alerts from the last k quarters as one sample, refreshed quarterly. A half-window stability check replaces consecutive-quarter tests.
ATL / BTL
Above / below the line. This KRI reads ATL history; any actual raise is validated by BTL sampling at the proposed new threshold, since observed data cannot show risk that never alerted.
Productive / FP
Productive (TP): alert with an investigative outcome. FP: reviewed and closed with none — the workload this KRI exists to reduce.

    4Guardrails and governance

    • Risk tier caps α. High-risk typologies run at α = 0 regardless of the dial — the tier selector above enforces this live.
    • Recency veto. A productive near the threshold in the most recent quarter blocks the candidate even when the pooled quantile clears h_min. Set "most recent productive at" below h_min to see the veto engage.
    • Volume floor. R must clear r_min so the saved hours pay for the tuning cycle.
    • BTL validation. This KRI reads observed ATL history only; it cannot see risk that never alerted. Every fired candidate goes to below-the-line sampling at the proposed new threshold before any change is implemented.
    • Small samples stay conservative. With fewer than 5 productives, H is flagged indicative and routed to analyst review; with 0–1 productives the AD routes to retirement / BTL review — the highest-impact cases, handled through a separate action rather than dropped.

    Positioning: the KRI identifies candidates and sizes the prize. Authorization of any raise stays with the existing tuning/BTL process. That separation keeps it a prioritization instrument, defensible to Model Risk without entering threshold-setting model scope.
